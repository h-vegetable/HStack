应用层详解 — apps/
apps/main-app
micro-app 基座 (Host Application)
整个平台的壳应用，基于 micro-app 实现微前端架构，负责全局布局、路由分发、子应用注册与挂载。
管理全局状态（用户信息、主题切换、侧边栏状态等），提供统一的 Header/Sidebar/Footer 布局。
ReactVitemicro-app 宿主全局布局
apps/portal
管控中心子应用
平台的门户与管控中枢，三个子页面边界清晰：
dashboard — 应用入口宫格，展示并跳转各子应用；
changelog — 各应用版本迭代与变更记录；
stats — 独立数据看板页，汇总各应用的关键指标（不与 dashboard 混在一起）；
未来可扩展博客/公告类功能。
ReactVitemicro-app 子应用门户数据看板
apps/mock-platform
Mock 平台子应用
根据后端接口文档一键生成 Mock 数据。
文档导入 — 支持 Swagger/OpenAPI JSON、Markdown 格式解析；
在线 Mock — 生成后直接在线使用，提供临时 Mock 服务地址；
离线下载 — 导出为 JSON/JS Mock 文件，可在本地项目中使用。
内置 Swagger/MD 解析引擎，自动提取接口路径、参数、响应结构。
ReactVitemicro-app 子应用SwaggerMock
apps/house-hunting
看房日志子应用
记录每次看房的详细信息与主观评价。
看房记录 — 小区、户型、价格、面积、位置等字段；
评分与标签 — 多维度打分 + 自定义标签；
统计对比 — 多房源横向对比、价格趋势、综合排名。
ReactVitemicro-app 子应用生活记录
apps/component-hub
组件中心子应用
统计、归纳、展示可用组件。
分类目录 — 按类型/功能/来源分类浏览；
在线预览 — 实时渲染组件 Demo，支持交互调试；
下载使用 — 一键复制安装命令或下载源码，支持 npm 包和源码两种方式。
⚠ 与 shared-ui 的边界：component-hub 不重复实现组件，而是消费 shared-ui 导出的 metadata（导出清单 + JSDoc + Demo 文件），自动渲染目录与预览，避免双倍维护。
ReactVitemicro-app 子应用shared-ui 消费方Storybook
apps/api
NestJS 后端服务
统一后端 API，采用模块化架构，TypeORM 连接 PostgreSQL（JSONB 字段对 Mock 数据 / 看房标签等动态结构友好）。
包含 Auth（认证）、User（用户）、Portal（管控）、Mock（解析生成）、HouseHunting（看房）、ComponentHub（组件）等业务模块。
提供统一的鉴权守卫、异常过滤、日志拦截、Sentry 上报等公共能力，并通过 @nestjs/swagger 暴露 /api/docs，可被 Mock 平台直接消费。
NestJSTypeORMPostgreSQLJWTSwagger

共享 TypeScript 类型 — 仅放横切类型
只收纳真正全局共享的类型，遵循"删掉某个子应用后该类型是否仍有用"的原则。

TypeScript横切类型前后端共享就近原则

类型分层策略

并非所有 Type 都应放进 shared-types

原则：横切类型进 shared-types，领域类型就近定义
判断标准："如果删掉某个子应用，这个类型还有存在的必要吗？"

✅ 进 shared-types 的横切类型：API 响应信封、分页结构、Auth/JWT 类型、通用枚举（状态/排序方向）、用户基础信息。

❌ 不进 shared-types 的领域类型：Portal 的统计 DTO、Mock 平台的 Swagger 解析类型、HouseHunting 的房源模型、ComponentHub 的组件元数据。
这些类型留在 apps/api/src/modules/{domain}/dto/，前端子应用通过 tsconfig paths alias 直接引用。
shared-types
api.d.ts — 响应信封
auth.d.ts — JWT Payload / 角色
common.d.ts — 通用枚举
所有应用都可能用到的类型。
API 领域 DTO
modules/portal/dto/ — 统计 & 更新日志
modules/mock/dto/ — Mock 项目 & 接口
modules/house-hunting/dto/ — 房源模型
前端通过 alias 引用，不污染 shared 包。
前端子应用消费方式
子应用 tsconfig 配置 path alias：
"@api/portal": ["../api/src/modules/portal/dto"]
然后直接 import { StatsDto } from '@api/portal'
⚠ DTO 跨包引用注意：保持「纯 type」
前端通过 alias 引 apps/api/src/modules/{domain}/dto 时，DTO 文件内不要直接写装饰器（@IsString()、@ApiProperty() 等）—— 会把 NestJS / class-validator 拖到前端 bundle 里，tree-shake 也救不回来。

建议拆成两文件：
• xxx.dto.ts — 纯 interface / type，前后端共享；
• xxx.validator.ts — class + 装饰器，仅后端用，implements 上面的 interface 保证字段同步。
结论：shared-types 只放横切类型，避免"一个类型改动 → 所有应用重建"。
领域类型就近定义在 API 模块内（保持纯 type，校验器分文件），前端通过 tsconfig alias 消费，既保持类型安全，又解耦了不必要的依赖。
共享样式架构 — 新增 shared-styles

为什么需要独立的样式基础包？

问题：shared-ui 承载了两件事
当前 shared-ui 同时包含 组件库 和 主题/样式基础。
如果某个子应用只想引用颜色 Token 或 CSS 变量，不需要引入整套组件库。
设计 Token 应该是一个零依赖的纯数据层，甚至可以被文档站、CLI 工具复用。
方案：新增 shared-styles，层级依赖
shared-styles（零依赖）→ shared-ui（依赖 shared-styles）→ apps/*（依赖两者）

shared-styles 提供：设计 Token (colors.ts / spacing.ts / typography.ts)、CSS Reset、CSS 变量定义、Tailwind 预设、HeroUI 主题覆盖。
子应用只需 import '@hstack/shared-styles/css/variables.css' 即可获得全局 CSS 变量。
包依赖关系：

shared-styles ← 零依赖，纯数据 + CSS
    ↓
shared-ui ← 依赖 shared-styles 的 Token 和 CSS 变量来构建组件
    ↓
apps/* ← 直接依赖 shared-styles（全局样式入口）+ shared-ui（组件）

修改品牌色只需改 tokens/colors.ts 一处，全局生效；
任何一个应用都可以独立引入样式基础，不强制依赖 React 组件。
状态管理策略 — Zustand

选型与跨 Shadow DOM 共享思路

为什么选 Zustand
无 Provider — 不需要包一层 Context，天然规避「React Context 不穿透 Shadow DOM」的问题；
API 极简 — create() 一个 store，useStore(selector) 即用即取；
体积小 — gzip 后 < 1KB，每个子应用各自打包也不心疼；
可订阅外部事件 — 适合接 micro-app 的 data 通道。
局部 store
各子应用 src/stores/ 下的业务态（页面筛选、表单草稿等），完全独立。
全局 store (基座)
main-app 持有 useGlobalStore：用户、主题、语言、侧边栏折叠态。
通过 <micro-app data={globalState}> 下发到子应用。
子应用「镜像 store」
子应用内部建一个轻量 store，订阅 window.microApp.addDataListener，把基座下发的数据同步进来；本地组件统一从镜像 store 读，独立运行时降级为读 localStorage。
核心约定：子应用从不直接从 React Context / window 读全局态，一律走自身的 Zustand 镜像 store —— 这样独立运行 / 嵌入运行只切换数据源，业务代码零改动。
微前端通信机制 — micro-app data 桥

跨 Shadow DOM 的状态与事件流

问题：React Context 不穿透 Shadow DOM
shared-auth 的 AuthProvider 写在基座，子应用即便引用同一个 React 实例，Context 也不会跨越 Shadow Root 边界。
因此 useAuth() 不能直接消费基座的 Context，必须走 micro-app 自己的通信通道。
方案：data props 下行 + customEvent 上行
下行（基座 → 子应用）：基座 <micro-app data={{ user, token, theme, locale }} />，子应用 window.microApp.addDataListener(cb) 订阅。
上行（子应用 → 基座）：子应用 window.microApp.dispatch({ type: 'auth:logout' })，基座监听 data-change 事件处理。
shared-auth 适配：useAuth() 内部检测 window.__MICRO_APP_ENVIRONMENT__，自动选择数据源（嵌入 → microApp data；独立 → localStorage + 接口）。
统一通信约定：
1. 所有跨应用状态走 micro-app 的 data 通道，不绕道全局变量；
2. 事件命名采用「领域:动作」格式（如 auth:logout、theme:change）；
3. shared-auth / shared-i18n 内部封装好检测逻辑，业务代码无感。
路由协同策略

主应用与子应用的路由如何分工

层级 路由策略 说明
main-app（基座） BrowserRouter + 一级路由 负责 /portal、/mock、/house、/components 一级分发，每个一级路由挂载对应 micro-app 标签。
子应用（嵌入态） BrowserRouter + basename 注入 basename 由基座通过 micro-app data 传入（如 /portal），子应用读取后用作 BrowserRouter basename。
子应用（独立态） BrowserRouter + basename = '/' 未检测到 micro-app 环境时退回根路径，独立 dev 体验完整。
keep-alive 模式
切换子应用时使用 <micro-app keep-alive />，子应用实例不销毁、路由状态保留，回切后停留在原页面。
history 同步
子应用内部 navigate('/list') 实际指向 /portal/list；浏览器前进后退由 micro-app 自动同步。
404 fallback
Nginx 配置 try_files $uri /index.html，所有未匹配路径回到基座 SPA，由基座路由判断是否为合法子应用前缀。
结论：basename 走 micro-app data 注入，子应用代码完全不感知自己被嵌在何处；基座统一管理 keep-alive / history / 404 fallback，子应用只关心自己 basename 内的路由。
鉴权方案讨论

鉴权写在公共部分还是每个子应用各自写？

方案 A：鉴权写在公共部分 ✓ 已采纳
将鉴权逻辑抽取到 packages/shared-auth，由基座应用 (main-app) 在顶层注入 AuthProvider。
子应用通过 useAuth() Hook 消费用户状态，通过 AuthGuard 组件保护路由。

优点：一次实现、所有子应用复用；用户状态全局统一（登录/登出所有子应用同步）；切换子应用无感知。
缺点：子应用与基座产生耦合，独立运行时需 mock Provider。
方案 B：每个子应用各自实现鉴权 (未采纳)
每个子应用独立管理自己的鉴权逻辑和路由守卫。

优点：子应用完全独立，可脱离基座单独运行。
缺点：重复代码多；登录状态不同步；每个子应用都要维护 token 刷新逻辑；改动需同步多处。
已采纳方案 A，采用「共享包 + 基座注入 + 子应用消费」模式：

1. packages/shared-auth 提供鉴权的所有抽象：AuthProvider、AuthGuard、useAuth、token 管理。
2. apps/main-app 在根组件注入 AuthProvider，持有全局鉴权状态。
3. 子应用通过 micro-app data 通道获取用户信息（不直接消费基座 Context），useAuth() 内部封装好这层适配。
4. 运行模式自动检测：useAuth() 检测 window.__MICRO_APP_ENVIRONMENT__：
    • AUTH_MODE = 'embedded' → 读 microApp data，token 刷新由基座负责，401 跳基座登录页；
    • AUTH_MODE = 'standalone' → 读 localStorage，自己调 /auth/refresh，401 跳本地登录页。
5. 子应用独立开发时，提供一个 MockAuthProvider 注入伪用户，方便本地调试。

既保持了代码复用，又不失子应用的独立开发与独立部署能力。

逐方案分析

qiankun — 不推荐
最大问题：和 Vite 不兼容。要求子应用暴露 UMD 格式生命周期，而 Vite 默认产出 ESM，需额外插件 (vite-plugin-qiankun) 做转换，坑多且维护滞后。
社区活跃度下降，阿里内部也在转向其他方案。全栈 Vite 项目强行适配纯属给自己找麻烦。
micro-app — 已选用
接入最简单：基座用 <micro-app name="portal" url="http://localhost:3001" /> 一行标签即可挂载。
Vite 原生支持 (@micro-app/vite-plugin)，Monorepo 适配好，沙箱够用。
⚠ HeroUI 兼容注意：HeroUI 的 Modal/Popover/Dropdown 等 Portal 组件默认挂载到 document.body，会逃逸 Shadow DOM 导致样式丢失。需在 shared-ui 中统一封装，用 container prop 指定挂载点为 Shadow Root 内节点。
wujie — 偏重
隔离最强 (iframe 级别)，适合对接老项目、异构技术栈。
但本项目所有子应用技术栈统一 (React + Vite)，iframe 隔离反而是负担：通信成本高、弹窗/路由同步麻烦。
Module Federation — 有代价
最契合 Monorepo (共享依赖天然解决)，性能最好 (构建时方案)。
但 Vite 支持不够成熟；子应用无法真正独立运行；版本冲突处理复杂；调试困难。适合更大规模的项目。
已选用 micro-app，核心理由：

1. Vite 兼容性最好 — 零折腾，官方提供 Vite 插件。
2. 接入成本最低 — 一个人维护所有子应用，效率优先。
3. 沙箱隔离够用 — 技术栈统一 (React + Vite)，不需要 iframe 级隔离。
4. 独立开发体验好 — 每个子应用可单独 pnpm dev 运行，配合 MockAuthProvider 不依赖基座。
5. 社区稳定 — 京东内部大规模使用，维护活跃。

本项目已有 shared packages 实现依赖共享，无需 Module Federation 的构建时共享能力。当前阶段 micro-app 是性价比最高的选择。
微前端共享依赖策略

shared packages 是各自打包还是基座统一提供？

策略 做法 优点 缺点
各自打包 ✓ 每个子应用独立 bundle shared-ui / shared-utils 等 子应用完全独立，可脱离基座单独开发 重复加载，总体积较大

已采纳「各自打包」策略。

核心理由：
1. 当前为个人项目，总体积可控，不需要极致优化。
2. 子应用独立开发体验优先 — pnpm dev 不依赖基座。
3. 未来若规模增长，可通过 shared-config 统一 external 配置一键切换。
